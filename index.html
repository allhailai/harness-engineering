What Is This?

A system for making AI agents reliable

Harness Engineering is the practice of encoding everything an AI agent needs to know directly into the repository — so it can work autonomously without constant human guidance.

When you first start using AI coding agents, you probably chat with them: “Hey, can you add a button here?” or “Fix this bug.” The agent helps, you review, you move on. This works fine for small tasks.

But as projects grow, this approach breaks down. Every new agent session starts cold — the agent has no memory of what you built yesterday, what rules you follow, or what mistakes were made last week. You end up re-explaining the same things over and over.

Harness Engineering solves this. Instead of relying on conversation to transfer knowledge, we encode that knowledge directly into the repository. The agent reads the repo, understands the project, and can work autonomously — following the right rules, avoiding known mistakes, and picking up tasks without hand-holding.

Phase 1

The Map — AGENTS.md

Give every AI agent a complete orientation to the project in a single file it reads at the start of every session.

🤔 Why This Was Built

Every AI agent session started completely cold — the agent had no idea what the project was about, which directories mattered, or what rules to follow. A human had to explain everything from scratch every single time. This was like hiring a new employee every day and re-onboarding them from zero.

The solution was AGENTS.md at the project root — a ~100-line “table of contents” for every AI agent. It covers everything the agent needs to orient itself instantly:

  • What the project is and how it is structured
  • Where the rules live (so agents know what to follow)
  • Key commands to run (so agents do not guess)
  • Architecture rules that must never be broken
  • What to do when a mistake happens
  • Known technical debt (so agents do not accidentally make it worse)

💡 Key Insight

Keeping AGENTS.md under 120 lines means the entire file fits in one AI context window read — the agent gets the full picture instantly. A longer file would get skimmed or truncated. Brevity is a feature, not a limitation.

Phase 2

Consolidate Rules Authority

One source of truth for all coding rules, automatically distributed to every AI tool.

🤔 Why This Was Built

Rules for how to write code existed in 4 different directories (.cursor/rules/, .agent/rules/, .roo/rules/, .gemini/). Each AI tool read from its own directory. Over time, these drifted apart — Cursor agents saw different rules than Roo agents. Updating a rule meant editing 4 files and hoping you did not miss one.

The solution was docs/rules/ as the single source of truth. A script (scripts/sync-rules.sh) generates all the tool-specific rule files automatically. Git hooks regenerate them whenever you pull or switch branches.

Think of it like a database with one source — you edit one file, and all tools get the update automatically. It is impossible to have drift because the generated files are never committed to git.

Phase 3

Deterministic Constraints

Turn architecture rules from documentation into automated tests that fail the build when violated.

🤔 Why This Was Built

The architecture rules (like “Core code must never import from the FH tenant module”) were documented but not enforced. An AI agent could violate them and nobody would catch it until a human reviewer noticed. Review comments for the same structural violations kept repeating.

The solution was automated tests that run in CI (Continuous Integration) and fail the build if architecture rules are violated. These tests make violations mechanically impossible to merge — no human review required.

🧪
Elixir Boundary Test
Scans all backend files to ensure Core modules never reference FH (tenant) modules. Fails CI if violated.
boundary_test.exs
⚛️
Frontend Boundary Test
Scans all frontend files to ensure core UI never imports from tenant UI. Enforces the same separation in TypeScript.
imports.test.ts
📦
API Envelope Test
Verifies all API responses are wrapped in the correct {data: ...} format. Catches structural regressions automatically.
api_envelope_test.exs
💡 Key Insight

“Mechanically impossible to merge” is stronger than “documented in a rule.” If the CI build fails, the violation cannot be merged — no human review required. Documentation tells people what to do; tests enforce it whether they read the docs or not.

Phase 4

The Failure Loop

Convert every agent mistake into a permanent rule that prevents that mistake from ever happening again.

🤔 Why This Was Built

When an AI agent made a mistake, the fix was applied to the code but rarely to the rules that would prevent the mistake from happening again. The same categories of errors kept recurring because there was no systematic process to convert failures into permanent prevention rules.

This is Mitchell Hashimoto’s core loop. The failure log is the history; the rules are the accumulated knowledge. Over time, the system gets smarter with every mistake.

1
Agent makes a mistake
The agent produces incorrect code, violates a convention, or makes a wrong assumption. This is normal and expected.
2
Fix the immediate issue
Correct the code. Do not just move on — this is the trigger for the improvement loop.
3
Log it in docs/failure-log.md
Record what happened, why it happened, and what rule would prevent it. This builds institutional memory.
4
Add or update a rule in docs/rules/
Write the rule that would have prevented the mistake. Be specific and prescriptive.
5
Run scripts/sync-rules.sh
Propagate the new rule to all AI tools automatically. Every agent now knows the rule.
6
That mistake never happens again
The rule is now part of the harness. Every future agent session starts with this knowledge already loaded.
Missing context Wrong assumptions Convention violations Boundary violations Redundant code Tool misuse
💡 Key Insight

The failure log is the history; the rules are the accumulated knowledge. Every mistake is an investment in the system getting smarter. A team that runs this loop for six months has a dramatically more capable agent harness than one that does not — because every mistake has been converted into prevention.

Phase 5

Garbage Collection

Fight entropy. Keep the harness accurate as the codebase evolves.

🤔 Why This Was Built

Documentation, rules, and architecture assertions rot over time. Code changes — files move, modules rename, features are deprecated — but the docs that reference them do not automatically update. Agents reading stale documentation produce wrong code. Without active maintenance, the harness itself becomes a source of confusion.

Three “garbage collection” agents fight entropy and keep the harness accurate as the codebase evolves:

📄
Doc Freshness Agent
A script that checks every file path referenced in documentation actually exists. Runs weekly in CI and on any PR that touches docs. Catches stale references before agents read them.
check-doc-freshness.sh
🏗️
Architecture Drift Agent
Checks that the actual codebase structure matches what is documented and tested. Verifies boundary tests are still comprehensive as the codebase grows.
drift detection
📊
Rule Coverage Agent
Monthly review that analyzes PR review comments for repeated patterns. If reviewers keep saying the same thing, it should become a rule.
monthly review
💡 Key Insight

The harness is only as good as its maintenance. Garbage collection prevents the system from degrading back into chaos over time. A harness without GC is like a garden without weeding — it looks fine at first, then slowly becomes unusable as stale information misleads agents.

Phase 6

Autonomous Execution

Enable agents to pick up and complete work end-to-end without human decomposition or hand-holding.

🤔 Why This Was Built

Even with all the previous phases, agents still needed humans to decompose work into tasks, explain what to build, and hand off between different agent modes. The goal was to move from “agent helps with coding” to “agent picks up and completes work autonomously.”

📋
Execution Plans
Structured documents in docs/plans/ that decompose features into ordered tasks with clear acceptance criteria. Agents read a plan and execute it end-to-end without human decomposition.
docs/plans/
📊
Quality Grades
Per-area quality grades (A/B/C/D) in docs/quality/ that tell agents where the code is solid (safe to build on) and where it is fragile (needs care or escalation).
docs/quality/
🤝
Handoff Protocol
A standard format for handing work between different agent modes (Architect → Code → Debug) so context is never lost between transitions.
handoff format

Plan lifecycle: Plans in Ready state can be picked up by agents autonomously. Plans in Draft state need human review first. This two-state system gives humans control over what agents can pick up while still enabling full autonomy on approved work.

Quality grades tell agents whether to proceed confidently (Grade A/B areas) or escalate to a human (Grade D areas). This prevents agents from building on fragile foundations without awareness.

💡 Key Insight

Plans in Ready state can be picked up by agents autonomously. Plans in Draft state need human review first. Quality grades tell agents whether to proceed confidently or escalate to a human. Together, these give you autonomous execution with human oversight exactly where it matters.
